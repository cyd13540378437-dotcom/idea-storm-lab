# 分析 Skill 优化手册

分析 Skill 的目标不是让回答“看起来很专业”，而是让团队更快判断一个想法是否值得继续验证。

当前 Skill 文件：

```text
skills/analysis_skill.md
```

## 核心原则

每次优化只改一个方向，不要一次改很多东西。

好的 Skill 变化应该让分析结果：

- 更具体
- 更敢指出风险
- 更少套话
- 更清楚地区分事实和假设
- 更能给出下一步验证动作
- 更适合你们团队的创业场景

## 推荐迭代流程

1. 选 3-5 条典型想法
   - 一个你觉得靠谱的想法
   - 一个非常模糊的想法
   - 一个你已经知道竞品很多的想法
   - 一个技术实现难度较高的想法
   - 一个商业模式不清楚的想法

2. 保存当前分析结果
   - 在页面里复制分析文本。
   - 或者先截图。
   - 重点记录你不满意的地方。

3. 只改 Skill 的一处
   - 例如只加强竞品分析。
   - 或者只加强风险质疑。
   - 或者只加强资本故事。

4. 重启服务

```bash
python3 app.py
```

服务器上则执行：

```bash
systemctl restart idea-storm-lab
```

5. 对同一条想法重新分析
   - 看是否更具体。
   - 看是否少了空话。
   - 看是否更符合你们讨论习惯。

6. 记录这次修改
   - 修改了什么。
   - 变好了什么。
   - 变差了什么。

## 常见优化方向

### 1. 分析太空

在 Skill 里加强：

```text
禁止使用“提升效率、优化体验、打造生态、赋能用户”等空泛表达，除非同时说明具体对象、场景和行为变化。
```

### 2. 竞品分析太弱

加强：

```text
竞品分析必须包含直接竞品、间接竞品和替代行为。无法确认具体竞品时，输出用户当前可能采用的替代方案，并说明应如何调研。
```

### 3. 太乐观

加强：

```text
每个想法必须指出最可能失败的原因，并给出可在 7 天内验证的动作。
```

### 4. 技术建议太虚

加强：

```text
技术选型必须分为“第一版验证足够用”和“后续规模化再考虑”，不要过早建议复杂架构。
```

### 5. 投资人视角太像广告

加强：

```text
资本视角必须同时包含增长故事和故事中最薄弱的一环，不允许只输出融资话术。
```

## 建议增加测试样例

可以新建：

```text
docs/analysis-test-cases.md
```

格式：

```text
## 样例 1：创业脑暴工具

输入：
...

我希望分析结果更关注：
- ...

我不希望出现：
- ...
```

当 Skill 越来越复杂时，这个样例集会很重要。它能防止你改好一个地方，又把另一个地方改坏。

## 推荐的 Skill 结构

当前先用一个综合 Skill。等你们真实使用一段时间后，可以拆成多个专家：

```text
skills/
  analysis_skill.md
  market_research_skill.md
  product_strategy_skill.md
  investor_story_skill.md
  validation_skill.md
```

但第一阶段不建议拆太早。现在最重要的是让一个综合 Skill 足够稳定。

## 判断 Skill 变好的标准

你可以用 1-5 分打分：

- 是否准确理解想法
- 是否指出关键用户和场景
- 是否能说出替代方案或竞品方向
- 是否能给出合理 MVP
- 是否能指出真实风险
- 是否能给出下一步验证动作
- 是否减少套话

每次修改后，只要平均分提升，就值得保留。
